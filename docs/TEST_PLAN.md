## 16. حالات الأخطاء (مرجع `docs/ERRORS.md`)

لكل خطأ مذكور في `ERRORS.md` يجب اختبار:
1. الرسالة للعميل واضحة وعربية.
2. الرسالة الإدارية تظهر في `/admin/errors` بتفاصيل كاملة.
3. log داخلي يُكتب.
4. الإجراء المقترح يعمل (retry/contact/…).
5. لا تضيع بيانات العميل (السلة/النموذج محفوظ).

**أمثلة حرجة:**
- payment.failed → السلة محفوظة + اقتراح وسيلة دفع أخرى.
- inventory.out → اقتراح بديل + إزالة من السلة باختيار العميل.
- shipping.unavailable → عرض شركات بديلة.
- network.lost → toast + retry تلقائي عند العودة.

## 17. خطة التنفيذ (Test Strategy)

### Pre-Release Checklist (لكل نشر)
1. **Smoke tests** على Preview (15 دقيقة): L1, S1, C1, K1, K7, SH1, N1.
2. **Regression** على ميزات لمسها التغيير (E2E).
3. **Security scan** عبر `security--run_security_scan` و`supabase--linter`.
4. **Lighthouse CI** ≥ 90 للأداء.
5. **Manual UAT** من فريق المنتج لأي تغيير UX.

### Automated Suites
- **Unit (U):** منطق `src/lib/*` (errors, seo, privacy, formatters).
- **Integration (I):** server functions + DB triggers + RLS.
- **E2E:** Playwright لمسارات K1, K7, K8, L1, S1, RT4.
- **Load (L):** k6 على `/api/v1/products` و`/checkout`.

### Production Smoke (بعد كل نشر — 5 دقائق)
- فتح الصفحة الرئيسية ✓
- بحث عن منتج ✓
- إضافة للسلة + checkout بمنتج اختبار + بطاقة اختبار ✓
- تأكيد وصول الإيميل ✓
- فتح `/admin/dashboard` كـ admin ✓

### تتبع الأخطاء
- جميع الأخطاء تُسجَّل في `error_logs`.
- مراجعة أسبوعية لأكثر 10 أخطاء تكراراً.
- SLA: P0 خلال ساعة، P1 خلال 24 ساعة، P2 ضمن السبرنت.
